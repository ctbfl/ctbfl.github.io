16:T1778,## 0. Preface

This article provides a concise reference for deploying the Pi0 series models.

We adopt a CS (Client-Server) architecture:

- Client: only handles robot action commands as input, and environment/robot feedback/observations as output (and runs the simulator/real world)
- Server: focuses on model inference, receiving feedback/observations from the client and outputting new action commands.

So you will see that we use two virtual environments later:
one for model execution, one for the simulator, and they communicate with each other via web interfaces.

## 1. Clone the Project

```bash
git clone --recurse-submodules git@github.com:Physical-Intelligence/openpi.git
cd openpi
````

* Here `--recurse-submodules` means recursive cloning; it will clone all referenced sub-repositories in the repo.
* Unless otherwise specified, all commands below are executed in the project root directory.

## 2. Environment Setup

### 2.1 Install uv

* **uv** is a new Python package and environment manager developed by Astral (formerly the PyO3 team).
  Its features: extremely fast, all-in-one, and can replace the combination of `pip + venv (virtualenv) + pip-tools` (the speed comes from Rust implementation + local caching + parallelism).

```bash
conda create -n uv_envs python=3.10
conda activate uv_envs
pip install uv
```

* The steps above are mainly to create a â€œshellâ€ environment for running uv.
  This conda environment `uv_envs` itself will not be used to execute any project code; it only hosts the uv tool and lets you run uv inside it.

### 2.2 Install Main Project Environment (Server)

```bash
GIT_LFS_SKIP_SMUDLE=1 uv sync
```

* `openpi/` is the project root, which contains the project configuration file `pyproject.toml`.
  `pyproject.toml` is the modern unified configuration file for Python projects, used to declare dependencies, build system, and tooling, replacing scattered `setup.py` and `requirements.txt`.

* `GIT_LFS_SKIP_SMUDGE=1` is a temporary environment variable set for the current command only (only effective on this line), recognized by Git LFS.
  When enabled, `git clone` or `git pull` will not automatically download large files managed by LFS, but only fetch their pointer files, which speeds up syncing and reduces bandwidth usage.
  If you later need the actual large binary files, you can manually run `git lfs pull` to download them.

* `uv sync` will create a `.venv` virtual environment folder under the project directory and install all required packages based on `pyproject.toml`.

```bash
source .venv/bin/activate
```

This activates the project-level virtual environment created by uv.
At the Python level, this environment completely replaces the previously activated conda environment.
Even if you see two environment names in your shell prompt, only the most recently activated one is effective.

![image-20251128204035983](/blog/reproducing-pi0-5-simulator/image-20251128204035983.png)

> ğŸ§© When a venv is stacked on top of a conda environment:
>
> * The Python executable is completely replaced.
> * Pythonâ€™s import path depends on the configuration inside the Python executable, so all available packages now come from the new venv.
> * The system `$PATH` is stacked, not replaced, so you can still use the `uv` command in the new environment â€” the system will find the `uv` executable from the conda environmentâ€™s path.
> * When you run `python`, the system searches `PATH` from front to back; since the latest environment is at the front, it will use the most recently activated Python environment.

### 2.3 Install LIBERO Environment (Client)

Since we will first do inference tests in the LIBERO environment, which has its own dependencies, it is better to create a separate virtual environment for it.

```bash
conda activate uv_envs # if you are not already in this conda environment
uv venv --python 3.8 examples/libero/.venv
source examples/libero/.venv/bin/activate
```

We first create a new virtual environment under the `examples/libero` folder.

Then we activate this new virtual environment, so you should see `(.venv)` appearing in front of `(uv_envs)` in your prompt.

Next, install the dependencies:

```bash
uv pip sync exmaples/libero/requirements.txt third_party/libero/requirements.txt --extra-index-url https://download.pytorch.org/whl/cu113 --index-strategy=unsafe-best-match
```

* `uv pip sync` is a uv command that synchronizes the virtual environmentâ€™s libraries to exactly match your `requirements.txt` files.
  It also supports multiple requirements files at once (it will merge them and try to find the best solution).
* The last two parameters are for compatibility with the specified PyTorch CUDA wheels.

Next, install submodules as local editable packages:

```bash
uv pip install -e packages/openpi-client
uv pip install -e third_party/libero
```

LIBERO also requires an environment variable:

```bash
export PYTHONPATH=$PYTHONPATH:$PWD/third_party/libero
```

* You can test with `python -m libero`.

## 3. Simple Run

First start the server:

```bash
source .venv/bin/activate # make sure you are in the server environment
python scripts/serve_policy.py --env LIBERO
```

* Here `--env` selects one of the pretrained policy weights provided by Physical Intelligence. Available options: `ALOHA`, `ALOHA_SIM`, `DROID`, `LIBERO`.

* In the [source code](https://github.com/Physical-Intelligence/openpi/blob/3e0325a5e6032a9ab6beef7395fe47ca724980a5/scripts/serve_policy.py), the default command-line arguments are defined in class `Args`, and `tyro.cli(Args)` is used to merge user-provided CLI arguments with these defaults.

  ![image-20251128220831692](/blog/reproducing-pi0-5-simulator/image-20251128220831692.png)

Then open another terminal (keeping the original server terminal running) and start the client:

```bash
python examples/libero/main.py
```

This example script will automatically run a test. After waiting for a while, you will get a video showing the execution result.
